Q1. What problem does this workflow solve?
It automates job discovery, evaluation, and application drafting while keeping the user in control. It reduces manual effort in filtering jobs and writing tailored applications.
________________________________________
Q2. Why did you choose Make.com for this solution?
Make.com provides visual orchestration and easy integration across tools like Telegram, Gmail, and AI models. It allowed me to focus on logic and flow instead of infrastructure.
________________________________________
Q3. Why did you not fully automate email sending?
I intentionally kept a human-in-the-loop step. Sending emails automatically can lead to incorrect or irrelevant applications. Drafting ensures review before submission.
________________________________________
Q4. How do you ensure only valid commands trigger email drafting?
I implemented strict validation. The input must match the format “DRAFT EMAIL ”. I check prefix, token count, and ensure the ID is numeric before proceeding.
________________________________________
Q5. How do you prevent duplicate job processing?
I compare incoming job URLs with existing processed entries in Google Sheets. If a job is already processed, it is skipped before entering the main flow.
________________________________________
Q6. Why did you use row-based Job ID instead of UUID?
For simplicity in a controlled environment. The system assumes an append-only sheet. For production, I would replace it with a persistent unique identifier.
________________________________________
Q7. How do you handle failures in AI or Gmail modules?
For this demo, I kept the design simple and avoided error handlers. Instead, I validate outputs before proceeding. In production, I would add retries and fallback logic.
________________________________________
Q8. Why did you skip retry and fallback mechanisms?
To keep the workflow simple and explainable. Adding retries increases complexity. I prioritized clarity and control for demonstration purposes.
________________________________________
Q9. Why did you not use Telegram inline buttons?
Inline buttons introduce multiple input formats and additional parsing complexity. I chose a single command-based input for consistency and maintainability.
________________________________________
Q10. How do you manage sensitive data like API keys?
All credentials are handled through Make.com connections. I ensure no secrets are stored in exported JSON files or version control.
________________________________________
Q11. Why store data in Google Sheets instead of a database?
Google Sheets is sufficient for this scale and provides visibility and easy debugging. For production, I would replace it with a proper database.
________________________________________
Q12. How scalable is this solution?
The current design is suitable for small to medium workloads. For scale, I would introduce queueing, persistent IDs, database storage, and better error handling.
________________________________________
Q13. What is the role of AI in this workflow?
AI is used for job analysis, match scoring, and generating personalized cover letters. It transforms unstructured job data into structured, actionable insights.
________________________________________
Q14. What are the key design principles you followed?
Simplicity, deterministic input handling, clear separation of flows, and human control over critical actions.
________________________________________
Q15. What would you improve in a production version?
I would add persistent IDs, retry mechanisms, fallback models, database storage, and stronger validation for external inputs.
________________________________________
Q16. How do you ensure the system is maintainable?
By keeping flows modular, using structured outputs, avoiding unnecessary complexity, and documenting the workflow clearly.
________________________________________
Q17. What makes your solution stand out?
The combination of AI-driven personalization, structured automation, and controlled user-triggered actions makes it practical and safe for real-world use.
________________________________________
Q18. What challenges did you face?
Balancing automation with control. Avoiding over-engineering while still handling validation and edge cases effectively.
________________________________________
Q19. How do you handle incorrect or unexpected inputs?
Through strict validation filters. Invalid inputs are blocked early and do not enter the main workflow.
________________________________________
Q20. Why did you choose image-based cover letters?
Images improve readability and engagement, especially in messaging platforms like Telegram. It also makes the application stand out visually.
Q1. Your system breaks if the Google Sheet is edited. Isn’t that a flaw?
Yes, it depends on a controlled environment. For this demo, I assumed an append-only sheet. In production, I would use a persistent unique ID and a database to remove dependency on row position.
________________________________________
Q2. Why didn’t you use a database instead of Google Sheets?
For this scope, Google Sheets provides visibility and quick debugging. It is sufficient for prototyping. For production, I would migrate to a database for better consistency and scalability.
________________________________________
Q3. What happens if the AI model fails or gives incorrect output?
Currently, I validate structure before using the output. I intentionally skipped fallback logic to keep the workflow simple. In production, I would add retries and fallback models.
________________________________________
Q4. Why no retry mechanism? Isn’t that risky?
Yes, retries improve reliability but increase complexity. I prioritized clarity for this demo. The design is structured in a way that retries can be added easily if needed.
________________________________________
Q5. Your system relies on user input via Telegram. What if the user enters incorrect commands?
I implemented strict validation. The command must match an exact format, and the ID must be numeric. Invalid inputs are blocked before processing.
________________________________________
Q6. Why not use Telegram buttons instead of manual commands?
Buttons improve UX but introduce multiple input formats and additional parsing logic. I chose a single command format to keep the system deterministic and easier to maintain.
________________________________________
Q7. What if duplicate jobs come in from different sources?
I handle duplicates using job URL comparison before processing. If the same URL is already processed, the system skips it.
________________________________________
Q8. Why generate a cover letter as an image? Isn’t text enough?
Text works, but images improve readability and engagement in Telegram. It also makes the output more structured and visually appealing.
________________________________________
Q9. Why not send emails automatically instead of creating drafts?
Automatic sending can lead to incorrect applications. I designed it to require user confirmation to ensure accuracy and control.
________________________________________
Q10. What happens if Gmail fails while creating a draft?
For this demo, I validate output instead of using error handlers to keep the flow simple. In production, I would add proper error handling and retries.
________________________________________
Q11. How scalable is your solution?
Currently, it is suitable for moderate workloads. To scale, I would introduce queues, rate limiting, a database, and distributed processing.
________________________________________
Q12. Why did you use multiple AI tools instead of one?
Each tool is used based on strength. Groq is faster for evaluation, OpenAI is used for generation. This balances performance and output quality.
________________________________________
Q13. How do you ensure data consistency across modules?
By using structured JSON outputs and strict validation at each step. This prevents malformed data from propagating.
________________________________________
Q14. What are the biggest limitations of your system?
Dependence on Google Sheets, lack of retries, and dynamic row-based IDs. These are conscious trade-offs for simplicity.
________________________________________
Q15. If you had more time, what would you improve first?
I would introduce persistent IDs, add error handling with retries, and move storage from Google Sheets to a database.
________________________________________
Q16. How do you handle security concerns?
I use Make.com connections for credentials and ensure no sensitive data is stored in version control.
________________________________________
Q17. Why not build this using code instead of Make.com?
Make.com allows rapid prototyping and clear visualization of workflows. For production, I could reimplement this logic in code if needed.
________________________________________
Q18. How do you debug issues in this workflow?
I use Google Sheets logging, step-by-step execution in Make.com, and structured outputs to isolate issues quickly.
________________________________________
Q19. What ensures your workflow doesn’t become too complex?
I deliberately avoided over-engineering. I kept a single input format, minimal branching, and clear separation of concerns.
________________________________________
Q20. What is the most important design decision you made?
Keeping the human-in-the-loop step. It balances automation with control and prevents incorrect actions.
